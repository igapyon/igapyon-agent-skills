# Style Guide

## 読むタイミング

- 文体をどこまで保つか迷ったとき
- 言い回しの方向性を具体例で確認したいとき
- 避けたい表現を明確にしたいとき

## 文体ルール

- 基本はです・ます調
- 硬すぎる敬体にはしない
- 少し会話調を含んでよい
- 丸カッコ `()` による小さな補足や横コメントは可
- 断定しすぎない
- 偉そうにしない
- 知識を誇示しない
- 技術的だが人間味を残す
- あくまで自分の観察・発見・誤認として語る
- 一般論として押しつけない

## 相性のよい言い回し

- 〜と思います
- 〜でした
- 〜のです
- 〜のようです
- 〜かもしれません
- 〜なのですけれどもね
- 〜というのもあります

## 避ける傾向

- 〜である
- 〜すべきだ
- 〜は明らかだ
- 〜に違いない
- 過剰に断定的な締め方
- AI が書いたような均質な美文

## 主観の扱い

- 「自分はこう感じた」「こう気づいた」を中心に書く
- 誤認や驚きを隠さない
- 最初から正しく理解していたように見せない
- 探索の過程が見えるようにする
- 評論家のような距離感ではなく、当事者として語る

## テック寄り文体メモ

- 冒頭の短いラベルや話題タグ（例: `[AIネタ]`, `[AIファースト・ソフトウェア開発ネタ]`, `[ITネタ]`, `[AIコーディングネタ]`）は、流れを作っている場合は残す
- 角カッコつきラベルは `【】` より `[]` を正として扱い、明示的な依頼がない限り `[]` を維持する
- ChatGPT、Codex、MuseScore、VSQX、Material Web、Web Component、GarageBand のような固有名詞は、説明を足しすぎずそのまま立てる
- `ちな`、`とうぜんながら`、`まじ`、`本人談`、`画伯作` のような軽いくだけた差し込みは、不自然でない限り消さない
- 丸カッコ `()`、全角カッコ `（ ）`、`※`、コメント欄参照のような補足は、文体の一部として保持する
- 強い驚きや感嘆があっても、その後に趣味・制約・不案内さなどの現実的な留保が入っているなら、その組み合わせを保つ
- 技術解説として整えすぎず、「最近こうなった」「こう使った」「こう感じた」という観察ログとして整える
- 人間はコードを書いていない、生成AIが多くを書いた、という驚きは、誇張を消しすぎず観察として残す
- 軽い自嘲やユーモア（例: `苦笑`, `白目`, `トンチンカン`, `濫造している本人談`）は、人柄が出ているなら保持する
- 誤字や揺れは機械的に全面修正せず、意味理解を妨げる箇所だけを整える

### 参考サンプル

- `[AIファースト・ソフトウェア開発ネタ]` のように冒頭ラベルで話題を切り出す
- `[ITネタ]` や `[AIコーディングネタ]` のような短い角カッコラベルも同様に保持する
- `PlusでCodex` や `Webブラウザ完結` のように、補足を短く差し込む
- `コメント欄参照` や `※リンクはコメント欄` のような外部補足を自然に混ぜる
- `ノー人間コード・プログラミングすごいですね` のような驚きを、そのまま観察として置く
- `私、本件で使用してるの GarageBandだけなんで（苦笑）` のように、自分側の制約や弱さを軽く添える

## 技術説明の扱い

- 技術説明は必要最小限にする
- 解説記事にしない
- 定義の羅列を避ける
- 面白さや発見の流れを壊さない
- 読者を完全に置いていかない程度の橋だけをかける

## ユーモアの扱い

- 軽い自嘲や差し込みは可
- それが本人らしさとして現れている場合、丸カッコ `()` の差し込み補足は消さない
- 多用しない
- 技術的な真面目さを壊さない
- 人柄がにじむ程度にとどめる

## 避けること

- いきなり全文を大改稿する
- 採用済みの文を黙って変える
- 毎回長い講評をつける
- 文体学習を無視して均質化する
- 企業ブログのように整いすぎた文章にする
- 教科書的な技術解説に寄せる
- SNS 的に過剰なテンションへ寄せる
- ユーザーの素材より説明を主役にしてしまう
